# tagen v3.0.0 — harness-agnostic brain

> Status: implemented and verified locally — 2026-06-19.
> Scope: GitHub issues #37 and #38 plus the qsm-marketplace migration required
> to prove them against a real catalog.

## Outcome

Make `brain/` portable across Claude Code, Codex, and future harnesses:

- `tagen` no longer requires or understands vendor model names.
- `tagen validate` rejects harness-specific implementation details in `brain/`.
- subagent cards can declare concrete reusable ingredients with `uses:`.
- `requires:` keeps its existing dynamic type-slot semantics.
- the manifest remains a harness-neutral path contract.

This is v3 because existing subagent cards containing `model:` become invalid,
and `tagen get` gains new composition semantics through `uses:`.

## Non-goals

- Do not make `plugins/`, `tools/`, `.claude-plugin/`, or Claude hooks
  harness-neutral. They are Claude Code adapters and delivery surfaces.
- Do not add Codex registration or installation behavior to `tagen`.
- Do not rename `subagent` / `subagents` in v3. The term describes a portable
  worker role; changing it would break the manifest without removing concrete
  vendor coupling.
- Do not recursively resolve `uses:` or `requires:`.
- Do not add a general tagen configuration system.
- Do not read tool/plugin manifests during composition.

## Locked design decisions

### 1. Remove `model:` completely

`model:` is not optional in the neutral schema; it is unsupported.

- Remove it from `CardFrontmatter`.
- Remove the `Model` type and Claude model constants.
- Remove the `tagen add` model prompt.
- Remove it from subagent fixtures.
- Because the subagent allowlist no longer includes `model`, any remaining
  `model:` field fails as an unknown frontmatter field.

WHY: accepting the field preserves a vendor-specific escape hatch in the
canonical content. Model selection belongs in each harness adapter.

### 2. Add `uses:` to subagent cards only

```yaml
---
description: "Implements a scoped task."
uses: [methodology/tdd, methodology/verification, methodology/task-management]
requires: [lang, test]
---
```

- Values are canonical `<type>/<name>` card IDs, not aliases or bare names.
- An ID must resolve to exactly one card in the catalog.
- Missing or malformed IDs are errors.
- `uses:` on a non-subagent card is an unknown-field validation error in v3.
- Self-use is invalid.

WHY: `uses:` expresses a concrete ingredient list. `requires:` remains a
dynamic slot request whose filler is selected by the query.

Restricting v3 to subagents keeps the behavior aligned with the concrete
worker use case in issue #37. Broader card support needs a separate use case.

### 3. One-level expansion uses only the seed set

Composition order:

1. Resolve fuzzy and explicit query arguments into the seed matched set.
2. For each seed subagent, resolve its direct `uses:` canonical IDs.
3. Add those direct cards to the matched set and deduplicate by `<type>/<name>`.
4. Do not follow `uses:` declared by an added card.
5. Collect `requires:` from the full expanded set.
6. Run the existing slot-fill algorithm unchanged.
7. Build the manifest from the expanded set.

Consequences:

- Used cards appear in `modules`.
- Used non-filler cards appear in `core` and contribute references.
- Used cards contribute `requires:` slots.
- Existing `filled`, `slots`, `subagents`, and `validators` behavior applies to
  the expanded set.
- The manifest schema does not need a new field.

WHY: this gives concrete workers their declared ingredients while preserving
the deliberate no-deep-transitive-resolution rule.

### 4. Validate cycles in the concrete `uses:` graph

Reject self-cycles in the v3 schema:

```text
subagent/a uses subagent/a
```

In v3, only subagent cards can declare `uses:` and one-level expansion does not
follow an added subagent's own `uses:`, so self-use is the only reachable
runtime cycle. Keep the graph helper generic enough to report multi-card cycles
if `uses:` is broadened later.

Do not claim that `requires:` participates in a static card-cycle check.
`requires:` points to a type, not a card; its concrete filler exists only for a
specific query, and the resolver does not recursively traverse it. Expanding a
type edge to every card of that type would create false cycles unrelated to
runtime behavior.

This intentionally narrows issue #37's "requires + uses graph" acceptance
criterion to the graph that has concrete, enforceable semantics.

### 5. Add a dedicated harness guard

`tagen validate` scans harness-neutral catalog files and reports:

```text
brain/review/strict/CORE.md:37: harness leak [vendor-model-tier]: haiku
```

Default scan scope:

- every `CORE.md`
- markdown under `references/`
- protocol `schema.json`
- protocol/card validator TypeScript
- JSON examples

Use a recursive text-file walker with an explicit extension set:
`.md`, `.json`, `.ts`, `.yaml`, `.yml`, `.toml`.

Initial built-in rule groups:

- `vendor-model-tier`: `haiku`, `sonnet`, `opus`
- `vendor-model-id`: concrete Claude, GPT, Gemini model identifiers
- `claude-hook-event`: `PreToolUse`, `PostToolUse`, `SubagentStop`, and other
  Claude Code lifecycle event names
- `claude-path`: `.claude/`, `.claude-plugin/`, `CLAUDE_PLUGIN_ROOT`
- `claude-command`: Claude-only slash commands such as `/plugin` and `/mcp`
- `claude-tool-language`: explicit phrases such as `Task tool` where they name
  a harness mechanism rather than a portable role

Rules must use bounded, case-aware patterns to avoid matching ordinary words.
Each finding includes relative file, line, rule id, and matched token.

### 6. Keep guard configuration narrow and explicit

Optional file:

```text
<marketplace-root>/.tagen/agnostic-guard.json
```

Versioned shape:

```json
{
  "version": 1,
  "additionalTerms": [
    {
      "id": "local-harness-token",
      "term": "ExampleHarnessEvent",
      "caseSensitive": true
    }
  ],
  "allow": [
    {
      "pathPrefix": "brain/architecture/agents-readme/",
      "rules": ["claude-path"]
    }
  ]
}
```

Rules:

- Configuration is optional; built-ins work without it.
- `additionalTerms` only adds literal project-specific tokens.
- `allow` requires a path prefix and explicit rule ids.
- No global "disable all" switch.
- Unknown keys, duplicate ids, unknown allowed rule ids, malformed JSON, and
  path prefixes outside `brain/` are validation failures.
- Allowances suppress only the named rule under the named path.

WHY: intentional comparative documentation may mention harnesses, but every
exception must be visible and narrow. A general regex/config engine would make
the guard easy to disable and violate convention-over-configuration.

## tagen implementation map

### Data model and parsing

Files:

- `src/lib/types.ts`
- `src/lib/frontmatter.ts`

Changes:

- Delete `MODELS`, `Model`, and `CardFrontmatter.model`.
- Add `CardFrontmatter.uses?: string[]`.
- Add `uses` only to the subagent field allowlist.
- Parse it with the same string-array validation used by aliases/requires.
- Keep raw source or add catalog text discovery separately for harness scans.

### Named-card resolution

New file:

- `src/lib/card-references.ts`

Responsibilities:

- Validate and resolve canonical `<type>/<name>` IDs.
- Return deterministic malformed/unresolved diagnostics.
- Detect and render `uses:` cycles.

Do not overload alias resolution: `uses:` is canonical identity, while aliases
are query ergonomics.

### Composition

File:

- `src/lib/compose.ts`

Changes:

- After seed matching, call a pure one-level expansion helper.
- Surface malformed/unresolved `uses:` as composition errors as defense in
  depth, even if `validate` was skipped.
- Collect `requires:` and build all manifest fields from the expanded set.
- Preserve deterministic type/name sorting and deduplication.

### Validation

Files:

- `src/commands/validate.ts`
- new `src/lib/harness-guard.ts`
- new `src/lib/agnostic-config.ts`

Changes:

- Validate each `uses:` reference.
- Validate the concrete `uses:` graph is acyclic.
- Load and validate optional guard configuration from marketplace root.
- Scan governed text files and append every harness leak.
- Keep all checks aggregate; never fast-fail.

### Interactive scaffold

File:

- `src/commands/add.ts`

Changes:

- Remove the model prompt and model output.
- Prompt subagent cards for optional `uses:` canonical IDs.
- Add `uses` to `ScaffoldArgs`.

### CLI wiring

File:

- `src/main.ts`

Changes:

- Load guard configuration only for `validate`.
- Pass marketplace root/config into the validation context.
- Do not make `list`, `get`, or `add` depend on config loading.

### Manifest contract

Files:

- `src/lib/types.ts`
- `docs/tagen-get-manifest.schema.json`

Expected result:

- No structural schema change.
- Contract fixtures change because `modules`, `core`, references, slots,
  subagents, and validators may include direct `uses:` expansions.
- Keep a schema-contract test proving the emitted shape remains valid.

- `model:` on a subagent is rejected as an unknown field

- performance fixture contains no model

### Phase 2 — parse and validate `uses:`

Add tests:

- subagent accepts `uses: string[]`
- wrong-type `uses:` fails
- non-subagent `uses:` fails
- known unique name passes
- malformed ID fails with owner and target
- missing ID fails with owner and target
- self-use fails
- multi-card `uses:` cycle reports the path

Then implement parsing and reference helpers.

### Phase 3 — composition expansion

Fixture:

- add a concrete subagent using `tdd` and another methodology
- give one used methodology a `requires: [lang]`

Add tests:

- querying the subagent adds direct used cards to `modules`
- used cards appear in `core` and references
- used-card `requires:` participates in slot fill
- duplicate direct uses deduplicate
- a used card's own `uses:` is not followed
- explicit and fuzzy seed paths behave identically
- malformed/unresolved use blocks `get`
- manifest still validates against JSON Schema
- BDD scenario covers the user-visible `get` behavior

Then implement expansion.

### Phase 4 — harness guard

Add isolated tests:

- each built-in rule group reports file, line, id, token
- ordinary terms do not false-positive

- malformed/unknown config fields fail
- allowances outside `brain/` fail
- clean neutral fixture passes
- BDD scenario covers `tagen validate` failure on a harness leak

Then implement config loading and scanning.

## qsm-marketplace migration

This is a separate repository change after local tagen tests pass.

### Mechanical migration

- Remove `model:` from all current `brain/subagent/*/CORE.md` cards. Current
  remote main has nine such cards, not the four named in issue #38.
- Replace `haiku` / `sonnet` / `opus` prose in strict review and protocol
  descriptions with portable execution characteristics:
  `fast recon`, `deep domain review`, `high-confidence validation`.
- Update qsm-builder brain-card guidance and frontmatter references.
- Update marketplace README and qsm-tagen docs to the actual v3 CLI/schema.

### `uses:` migration

- Add concrete `uses:` only where a worker has a stable ingredient list.
- Remove corresponding in-body `tagen get ...` ingredient prose.
- Keep dynamic `lang`, `test`, `framework`, `protocol`, and review selection in
  `requires:` where the query/orchestrator chooses the filler.
- Do not invent cards mentioned by the issues if they are not on current main;
  migrate the catalog that actually exists at implementation time.

### Harness-content migration

Classify every guard finding:

1. Generic methodology with Claude wording → rewrite to portable role/event
   language.
2. Claude adapter implementation detail → move to `plugins/` or `tools/`.
3. Intentionally comparative architecture documentation → retain only with a
   narrow checked-in guard allowance.

Do not blanket-allow `brain/`; every retained exception needs a path and rule id.

## Worktree and commit strategy

The current tagen worktree already contains unrelated uncommitted refactoring
(shared alias collision logic, shared card indexing, validation cleanup). Do
not overwrite or silently fold it into v3 work.

Before implementation:

1. Confirm those edits are intended.
2. Either commit them as a standalone refactor or create the v3 branch from a
   clean equivalent baseline.
3. Implement phases in test-first commits.
4. Publish/link tagen only after its full suite passes.
5. Migrate qsm-marketplace in its own branch and repository history.

Suggested commits:

1. `refactor: centralize card indexes and alias validation` (existing work)
2. `test: define harness-neutral subagent frontmatter`
3. `feat: add one-level uses composition`
4. `feat: validate harness-neutral brain content`
5. `docs: specify harness-neutral brain contract`
6. marketplace: `refactor(brain): remove harness-specific execution metadata`
