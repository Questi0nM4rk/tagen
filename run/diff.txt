+# tagen v3.0.0 — harness-agnostic brain
+
+> Status: implemented and verified locally — 2026-06-19.
+> Scope: GitHub issues #37 and #38 plus the qsm-marketplace migration required
+> to prove them against a real catalog.
+
+## Outcome
+
+Make `brain/` portable across Claude Code, Codex, and future harnesses:
+
+- `tagen` no longer requires or understands vendor model names.
+- `tagen validate` rejects harness-specific implementation details in `brain/`.
+- subagent cards can declare concrete reusable ingredients with `uses:`.
+- `requires:` keeps its existing dynamic type-slot semantics.
+- the manifest remains a harness-neutral path contract.
+
+This is v3 because existing subagent cards containing `model:` become invalid,
+and `tagen get` gains new composition semantics through `uses:`.
+
+## Non-goals
+
+- Do not make `plugins/`, `tools/`, `.claude-plugin/`, or Claude hooks
+  harness-neutral. They are Claude Code adapters and delivery surfaces.
+- Do not add Codex registration or installation behavior to `tagen`.
+- Do not rename `subagent` / `subagents` in v3. The term describes a portable
+  worker role; changing it would break the manifest without removing concrete
+  vendor coupling.
+- Do not recursively resolve `uses:` or `requires:`.
+- Do not add a general tagen configuration system.
+- Do not read tool/plugin manifests during composition.
+
+## Locked design decisions
+
+### 1. Remove `model:` completely
+
+`model:` is not optional in the neutral schema; it is unsupported.
+
+- Remove it from `CardFrontmatter`.
+- Remove the `Model` type and Claude model constants.
+- Remove the `tagen add` model prompt.
+- Remove it from subagent fixtures.
+- Because the subagent allowlist no longer includes `model`, any remaining
+  `model:` field fails as an unknown frontmatter field.
+
+WHY: accepting the field preserves a vendor-specific escape hatch in the
+canonical content. Model selection belongs in each harness adapter.
+
+### 2. Add `uses:` to subagent cards only
+
+```yaml
+---
+description: "Implements a scoped task."
+uses: [methodology/tdd, methodology/verification, methodology/task-management]
+requires: [lang, test]
+---
+```
+
+- Values are canonical `<type>/<name>` card IDs, not aliases or bare names.
+- An ID must resolve to exactly one card in the catalog.
+- Missing or malformed IDs are errors.
+- `uses:` on a non-subagent card is an unknown-field validation error in v3.
+- Self-use is invalid.
+
+WHY: `uses:` expresses a concrete ingredient list. `requires:` remains a
+dynamic slot request whose filler is selected by the query.
+
+Restricting v3 to subagents keeps the behavior aligned with the concrete
+worker use case in issue #37. Broader card support needs a separate use case.
+
+### 3. One-level expansion uses only the seed set
+
+Composition order:
+
+1. Resolve fuzzy and explicit query arguments into the seed matched set.
+2. For each seed subagent, resolve its direct `uses:` canonical IDs.
+3. Add those direct cards to the matched set and deduplicate by `<type>/<name>`.
+4. Do not follow `uses:` declared by an added card.
+5. Collect `requires:` from the full expanded set.
+6. Run the existing slot-fill algorithm unchanged.
+7. Build the manifest from the expanded set.
+
+Consequences:
+
+- Used cards appear in `modules`.
+- Used non-filler cards appear in `core` and contribute references.
+- Used cards contribute `requires:` slots.
+- Existing `filled`, `slots`, `subagents`, and `validators` behavior applies to
+  the expanded set.
+- The manifest schema does not need a new field.
+
+WHY: this gives concrete workers their declared ingredients while preserving
+the deliberate no-deep-transitive-resolution rule.
+
+### 4. Validate cycles in the concrete `uses:` graph
+
+Reject self-cycles in the v3 schema:
+
+```text
+subagent/a uses subagent/a
+```
+
+In v3, only subagent cards can declare `uses:` and one-level expansion does not
+follow an added subagent's own `uses:`, so self-use is the only reachable
+runtime cycle. Keep the graph helper generic enough to report multi-card cycles
+if `uses:` is broadened later.
+
+Do not claim that `requires:` participates in a static card-cycle check.
+`requires:` points to a type, not a card; its concrete filler exists only for a
+specific query, and the resolver does not recursively traverse it. Expanding a
+type edge to every card of that type would create false cycles unrelated to
+runtime behavior.
+
+This intentionally narrows issue #37's "requires + uses graph" acceptance
+criterion to the graph that has concrete, enforceable semantics.
+
+### 5. Add a dedicated harness guard
+
+`tagen validate` scans harness-neutral catalog files and reports:
+
+```text
+brain/review/strict/CORE.md:37: harness leak [vendor-model-tier]: haiku
+```
+
+Default scan scope:
+
+- every `CORE.md`
+- markdown under `references/`
+- protocol `schema.json`
+- protocol/card validator TypeScript
+- JSON examples
+
+Use a recursive text-file walker with an explicit extension set:
+`.md`, `.json`, `.ts`, `.yaml`, `.yml`, `.toml`.
+
+Initial built-in rule groups:
+
+- `vendor-model-tier`: `haiku`, `sonnet`, `opus`
+- `vendor-model-id`: concrete Claude, GPT, Gemini model identifiers
+- `claude-hook-event`: `PreToolUse`, `PostToolUse`, `SubagentStop`, and other
+  Claude Code lifecycle event names
+- `claude-path`: `.claude/`, `.claude-plugin/`, `CLAUDE_PLUGIN_ROOT`
+- `claude-command`: Claude-only slash commands such as `/plugin` and `/mcp`
+- `claude-tool-language`: explicit phrases such as `Task tool` where they name
+  a harness mechanism rather than a portable role
+
+Rules must use bounded, case-aware patterns to avoid matching ordinary words.
+Each finding includes relative file, line, rule id, and matched token.
+
+### 6. Keep guard configuration narrow and explicit
+
+Optional file:
+
+```text
+<marketplace-root>/.tagen/agnostic-guard.json
+```
+
+Versioned shape:
+
+```json
+{
+  "version": 1,
+  "additionalTerms": [
+    {
+      "id": "local-harness-token",
+      "term": "ExampleHarnessEvent",
+      "caseSensitive": true
+    }
+  ],
+  "allow": [
+    {
+      "pathPrefix": "brain/architecture/agents-readme/",
+      "rules": ["claude-path"]
+    }
+  ]
+}
+```
+
+Rules:
+
+- Configuration is optional; built-ins work without it.
+- `additionalTerms` only adds literal project-specific tokens.
+- `allow` requires a path prefix and explicit rule ids.
+- No global "disable all" switch.
+- Unknown keys, duplicate ids, unknown allowed rule ids, malformed JSON, and
+  path prefixes outside `brain/` are validation failures.
+- Allowances suppress only the named rule under the named path.
+
+WHY: intentional comparative documentation may mention harnesses, but every
+exception must be visible and narrow. A general regex/config engine would make
+the guard easy to disable and violate convention-over-configuration.

+export interface ConfigLoadResult {
+  config: AgnosticGuardConfig;
+  errors: string[];

+
+const CONFIG_PATH = [".tagen", "agnostic-guard.json"] as const;
+const TOP_LEVEL_KEYS = new Set(["version", "additionalTerms", "allow"]);
+const TERM_KEYS = new Set(["id", "term", "caseSensitive"]);
+const ALLOW_KEYS = new Set(["pathPrefix", "rules"]);
+
+export function loadAgnosticConfig(
+  root: string,
+  builtInRuleIds: ReadonlySet<string>
+): ConfigLoadResult {
+  const path = join(root, ...CONFIG_PATH);
+  const empty: AgnosticGuardConfig = { additionalTerms: [], allow: [] };
+  if (!existsSync(path)) return { config: empty, errors: [] };
+
+  let raw: unknown;
+  try {
+    raw = JSON.parse(readFileSync(path, "utf8"));
+  } catch (error) {
+    return {
+      config: empty,
+      errors: [`${CONFIG_PATH.join("/")}: invalid JSON: ${messageOf(error)}`],
+    };
+  }
+  if (!isRecord(raw)) {
+    return {
+      config: empty,
+      errors: [`${CONFIG_PATH.join("/")}: config must be a JSON object`],
+    };
+  }
+
+  const errors: string[] = [];
+  rejectUnknownKeys(raw, TOP_LEVEL_KEYS, "config", errors);
+  if (raw.version !== 1) errors.push("config: version must be 1");
+
+  const additionalTerms = parseTerms(raw.additionalTerms, builtInRuleIds, errors);
+  const knownRuleIds = new Set([
+    ...builtInRuleIds,
+    ...additionalTerms.map((term) => term.id),
+  ]);
+  const allow = parseAllowances(raw.allow, knownRuleIds, errors);
+
+  return { config: { additionalTerms, allow }, errors };
+}
+
+function parseTerms(
+  value: unknown,
+  reservedIds: ReadonlySet<string>,
+  errors: string[]
+): AdditionalTerm[] {
+  if (value === undefined) return [];
+  if (!Array.isArray(value)) {
+    errors.push("config: additionalTerms must be an array");
+    return [];
+  }
+  const out: AdditionalTerm[] = [];
+  const ids = new Set<string>();
+  for (const [index, item] of value.entries()) {
+    const tag = `config: additionalTerms[${index}]`;
+    if (!isRecord(item)) {
+      errors.push(`${tag} must be an object`);
+      continue;
+    }
+    rejectUnknownKeys(item, TERM_KEYS, tag, errors);
+    if (typeof item.id !== "string" || item.id.length === 0) {
+      errors.push(`${tag}.id must be a non-empty string`);
+      continue;
+    }
+    if (ids.has(item.id)) errors.push(`${tag}.id duplicates '${item.id}'`);
+    if (reservedIds.has(item.id)) {
+      errors.push(`${tag}.id conflicts with built-in rule '${item.id}'`);
+    }
+    ids.add(item.id);
+    if (typeof item.term !== "string" || item.term.length === 0) {
+      errors.push(`${tag}.term must be a non-empty string`);
+      continue;
+    }
+    if (typeof item.caseSensitive !== "boolean") {
+      errors.push(`${tag}.caseSensitive must be a boolean`);
+      continue;
+    }
+    out.push({
+      id: item.id,
+      term: item.term,
+      caseSensitive: item.caseSensitive,
+    });
+  }
+  return out;
+}

+function parseAllowances(
+  value: unknown,
+  knownRuleIds: ReadonlySet<string>,
+  errors: string[]
+): GuardAllowance[] {
+  if (value === undefined) return [];
+  if (!Array.isArray(value)) {
+    errors.push("config: allow must be an array");

+  }

+    }

+  }

+  for (const c of cards) canonical.set(c.id.name, c.id);
+
+  const seen = new Map<string, CardId>();
+  for (const c of cards) {
+    for (const alias of c.frontmatter.aliases ?? []) {
+      const collidingCanonical = canonical.get(alias);
+      if (collidingCanonical) {
+        errors.push(
+          `alias '${alias}' on ${cardKey(c.id)} collides with canonical name ${cardKey(collidingCanonical)}`
+        );
+      }
+      const prior = seen.get(alias);
+      if (prior) {
+        errors.push(
+          `alias '${alias}' collides between ${cardKey(prior)} and ${cardKey(c.id)}`
+        );
+      } else {
+        seen.set(alias, c.id);
+      }
+    }
+  }
+  return errors;

+/** Resolve canonical `<type>/<name>` IDs declared by one card. */
+export function resolveUses(owner: Card, cards: Card[]): UsesResolution {
+  const index = cardIndex(cards);
+  const resolved: Card[] = [];
+  const errors: string[] = [];
+  const seen = new Set<string>();
+
+  for (const target of owner.frontmatter.uses ?? []) {
+    if (!isCardKey(target)) {
+      errors.push(
+        `${cardKey(owner.id)}: invalid uses target '${target}', expected <type>/<name>`
+      );
+      continue;
+    }
+    if (target === cardKey(owner.id)) {
+      errors.push(`${cardKey(owner.id)}: uses cannot reference itself`);
+      continue;
+    }
+    const card = index.get(target);
+    if (!card) {
+      errors.push(`${cardKey(owner.id)}: unknown card in uses: ${target}`);
+      continue;
+    }
+    if (!seen.has(target)) {
+      seen.add(target);
+      resolved.push(card);
+    }
+  }
+
+  return { cards: resolved, errors };

+/** Expand only direct uses from the original seed set. */
+export function expandDirectUses(seeds: Card[], cards: Card[]): UsesResolution {
+  const expanded = new Map(seeds.map((card) => [cardKey(card.id), card] as const));
+  const errors: string[] = [];
+

+    const result = resolveUses(seed, cards);
+    errors.push(...result.errors);
+    for (const card of result.cards) expanded.set(cardKey(card.id), card);
+  }

+/** Detect cycles in the concrete uses graph and render each cycle once. */
+export function findUsesCycles(cards: Card[]): string[] {
+  const edges = new Map<string, string[]>();
+  for (const card of cards) {
+    const owner = cardKey(card.id);
+    edges.set(
+      owner,
+      (card.frontmatter.uses ?? []).filter((target) => isCardKey(target))
+    );
+  }
+
+  const visiting = new Set<string>();
+  const visited = new Set<string>();
+  const stack: string[] = [];
+  const cycles = new Set<string>();
+
+  const visit = (node: string): void => {
+    if (visited.has(node)) return;
+    if (visiting.has(node)) {
+      const start = stack.indexOf(node);
+      const cycle = [...stack.slice(start), node].join(" -> ");
+      cycles.add(cycle);
+      return;
+    }
+    visiting.add(node);
+    stack.push(node);
+    for (const next of edges.get(node) ?? []) {
+      if (edges.has(next)) visit(next);
+    }
+    stack.pop();
+    visiting.delete(node);
+    visited.add(node);
+  };
+
+  for (const node of [...edges.keys()].sort()) visit(node);
+  return [...cycles].sort().map((cycle) => `uses cycle: ${cycle}`);
+}

+  config: AgnosticGuardConfig

+  const rules = [
+    ...BUILT_IN_RULES,
+    ...config.additionalTerms.map((term) => ({

+      pattern: new RegExp(escapeRegExp(term.term), term.caseSensitive ? "gu" : "giu"),
+    })),

+      for (const rule of rules) {
+        if (isAllowed(rel, rule.id, config)) continue;

+        for (const match of line.matchAll(rule.pattern)) {
